01

Production RAG Pipeline

Hybrid search + re-ranking, sub-300ms retrieval, FastAPI backend

RAGFastAPIQdrantFAISSBM25Cross-Encoder

🔴 The Problem

Document Q&A systems in enterprise settings need accurate answers retrieved from large knowledge bases in real time. Off-the-shelf RAG setups using naive top-k cosine search regularly fail on keyword-heavy queries — like exact version numbers, product codes, or domain jargon — where semantic search alone doesn't cut it.

⚠️ Why Existing Solutions Fail

  • Pure dense retrieval misses exact-match queries (fails on IDs, codes, rare terms)
  • Pure BM25 misses paraphrase and intent-based queries
  • No re-ranking step → top retrieved chunk ≠ most relevant chunk
  • Naive chunking (fixed 512 tokens) breaks context boundaries

🏗️ System Architecture

1
Ingestion Pipeline — Documents chunked using sliding window with semantic boundary detection. Chunks embedded with sentence-transformers.
↓
2
Dual Index — Dense vectors stored in Qdrant (cosine similarity). Sparse BM25 index built in parallel for keyword retrieval.
↓
3
Hybrid Search — Query hits both indexes simultaneously. Results merged using Reciprocal Rank Fusion (RRF) to produce a unified candidate set.
↓
4
Re-ranking — Cross-encoder scores each candidate against the query. Top-3 chunks selected for context injection.
↓
5
LLM Generation — Structured prompt with retrieved context → LLM produces grounded answer via FastAPI endpoint.

🛠️ Tech Stack & Why

QdrantProduction-grade vector DB with filtering and payload support
FAISSFast local approximate nearest neighbour for dev/testing
BM25 (rank_bm25)Best sparse retriever for keyword matching without a search engine
Cross-EncoderMore accurate than bi-encoder for final ranking (accepts query+doc pair)
FastAPIAsync, typed, fast — production API standard for Python ML systems

⚡ Key Optimizations

  • Sliding window chunking with 20% overlap to avoid context splits
  • Async FastAPI endpoints to handle concurrent retrieval calls
  • Pre-computed BM25 index loaded at startup — no per-request rebuild
  • Cross-encoder only on top-20 candidates, not full index — critical for latency
  • Trade-off: Re-ranking adds ~40ms but increases answer precision significantly — kept it in

📊 Results

~0.2–0.3sEnd-to-end retrieval latency
HybridSearch beats pure dense by significant margin on keyword queries
FastAPIREST API ready for production integration
02

LLM-Based Automation System

Prompt pipelines, API orchestration, intelligent workflow automation

LLMsPrompt EngineeringAPI IntegrationWorkflow Orchestration

🔴 The Problem

Manual workflows involving data extraction, classification, transformation, and routing across multiple APIs are slow, error-prone, and don't scale. The challenge: replace a multi-step human workflow with an LLM-driven pipeline that is reliable, structured, and auditable.

⚠️ Why Existing Solutions Fail

  • Generic LLM calls without structured output → parsing failures in production
  • No retry or validation logic → silent errors downstream
  • Prompt templates without versioning → regression when model updates
  • No separation between LLM reasoning steps → hard to debug

🏗️ System Architecture

1
Input Ingestion — Raw data from external APIs / file sources normalized into a consistent schema.
↓
2
Prompt Pipeline — Multi-stage prompt chain: classify → extract → transform → validate. Each step has a defined input/output contract.
↓
3
Structured Output Parsing — LLM outputs constrained to JSON schema. Validation layer rejects and retries malformed outputs.
↓
4
Routing & Action Layer — Based on classification output, system routes to appropriate downstream API or action.

🛠️ What I Built

  • Designed modular prompt templates with versioning support
  • Built a retry wrapper with exponential backoff for LLM API calls
  • Implemented JSON schema validation on every LLM output before passing downstream
  • Created an audit log of every LLM decision for traceability
  • Integrated multiple third-party APIs into a single orchestrated workflow

⚡ Key Engineering Decisions

  • Trade-off: Single large prompt vs. chained smaller prompts — chose chaining for debuggability
  • Kept LLM out of routing logic — deterministic code handles control flow, LLM only extracts/classifies
  • Structured outputs with Pydantic models — type safety end to end
  • Stateless pipeline design — each step independent, easy to test in isolation

📊 Results

End-to-EndFully automated multi-step workflow
AuditableEvery LLM decision logged and traceable
RobustRetry + validation prevents silent failures
03

HistoGPT — Multimodal AI Optimization

Memory reduction, dimension compression, inference efficiency for pathology AI

Multimodal AIVision-LanguageInference OptimizationVector DBs

🔴 The Problem

HistoGPT is a vision-language model for histopathology — it processes whole-slide medical images (which are gigapixel in size) and generates diagnostic text. The challenge: the model's feature extraction stage produced massive, redundant feature vectors that made inference slow and memory-intensive, blocking deployment on realistic hardware.

⚠️ Why This Was Hard

  • Whole-slide images can't fit in GPU memory — must be processed as patches
  • Naive patch aggregation produces very high-dimensional feature tensors
  • Reducing dimensions aggressively risks losing clinically relevant features
  • No existing benchmarks for multimodal pathology model inference optimization

🏗️ What I Did

1
Memory Profiling — Profiled model inference end-to-end to identify memory hotspots in the feature extraction stage.
↓
2
Dimension Reduction — Applied targeted PCA and attention-based pooling to reduce feature vector dimensionality while preserving variance.
↓
3
Inference Efficiency — Optimized patch batching and reduced redundant forward passes through caching of extracted features.
↓
4
Vector DB Integration — Explored storing patch embeddings in a vector database for efficient retrieval-based diagnostic search across cases.

⚡ Key Optimizations

  • Reduced feature dimensionality while monitoring downstream generation quality
  • Cached patch features to avoid recomputing on repeated inference
  • Explored approximate nearest-neighbor search for similar-case retrieval
  • Trade-off: Aggressive compression risks diagnostic accuracy — I measured at each compression level and stopped at the Pareto-optimal point

🧠 What I Learned

  • Multimodal models have very different memory bottlenecks vs. text-only LLMs
  • Clinical domain requires extreme care in optimization — accuracy isn't negotiable
  • Vector databases for medical images open up retrieval-augmented diagnosis — a genuinely underexplored space

📊 Results

↓ MemoryReduced peak memory usage during inference
↓ DimsFeature dimensions reduced without quality loss
FasterImproved inference speed via batching & caching
